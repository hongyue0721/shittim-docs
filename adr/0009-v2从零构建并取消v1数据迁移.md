# ADR-0009：v2 从零构建并取消 v1 数据迁移

- 状态：accepted
- 日期：2026-07-20
- 实现状态徽章：**partial** — 唯一进度源：[`docs/IMPLEMENTATION_MATRIX.md`](../docs/IMPLEMENTATION_MATRIX.md) · [`docs/PROGRESS.md`](../docs/PROGRESS.md)
- 本 ADR 作废里程碑名 `V2ProductionWriteCutover`，替换为 `V2InitialBuildActive`

## 背景

仓库尚未对真实用户交付可连接 runtime。历史上为“从已有 v1 开发库平滑切到 v2”准备了数据迁移、legacy direct-child provenance、mixed Outbox production 兼容与 `V2ProductionWriteCutover` 原子切换谓词。这些假设把未发布实现当成了需要兼容的生产遗产，导致：

1. 必须维护 v1 write 路径、legacy reader/migration API 与 provenance 分支；
2. migration/reconciliation 与 cutover 谓词绑定 production bindings 启用时机，把初始交付伪装成“从旧部署切换”；
3. 存在自动补写、隐式升级或“读后补齐”半迁移风险。

owner 已拍板：没有用户，v2 不从 v1 迁移数据，从头 build。当前存储格式就是 v2 fresh baseline。

## 决策

### 1. 取消 v1 数据迁移；`V2InitialBuildActive` 取代 cutover

- 没有用户数据需要保留；v2 从零构建。
- 里程碑名 `V2ProductionWriteCutover` **作废**，替换为 `V2InitialBuildActive`。
- 当前权威存储格式即 v2 fresh baseline：不执行、也不支持 v1 业务数据迁移。
- 发现旧开发数据库（含仅含 v1 Task/Outbox/Approval 写路径产物、或无法被 v2 fresh baseline 识别的 ledger/shape）时，启动必须 **明确拒绝**，返回稳定可诊断的 `reinitialize-required` 类错误；禁止自动清库、隐式升级、读后补写或“尽量兼容打开”。
- production MethodVersionBindings、v2 dispatcher registration 与 v2 repository write gate 作为 **初始交付** 一并启用，不再以“cutover 时从空 bindings 原子切换旧部署”为叙事。

### 2. v1 runtime 写路径全部删除，而非加版本分支

实现切片必须删除下列 runtime 写能力，而不是在同一入口加 v1/v2 分支：

| 删除项 | 说明 |
|---|---|
| v1 `task.create` handler / adapter / repository write API | active KCP 路径对 v1 request 只得 `unsupported_schema_version` |
| legacy direct-child 写路径 | 不再有任何 production 写入口可直接写 `parent_task_id!=null` |
| `append_legacy_event_v1` | production store 不再提供 v1 Outbox append |
| `StoredEventEnvelope::LegacyV1` runtime variant | production store API 不再以 mixed legacy envelope 作为 active 读模型出口 |

active 写路径只承认 v2 合同对象。未发布的现有 v1 内部代码在实现切片中移除，不得以“兼容旧开发库”名义保留 production write。

### 3. 保留 vs 删除的精确边界

取消的是 **v1 数据兼容 / 迁移运行机制**，不是所有 retained Schema 资产：

| 保留 | 删除 / 禁止 |
|---|---|
| TaskSpec v1、TaskScope v1、TaskCreatedPayload v1 等仍被 v2 response/payload 引用的 retained Schema | v1 TaskCreate production write、direct-child write |
| EventEnvelope v1 / CausationRef v1 的 Schema source 与历史兼容 fixture | EventEnvelope v1 / CausationRef v1 的 production store API |
| retained EventPollResponse v1 的既有 wire（在 poll 升级前） | 用 poll v1 返回 EventEnvelope v2；伪造无 owner 的 producer |
| `TaskCreationProvenance` 的 `root_command_v2` / `child_action_v2` | `legacy_direct_create_v1`、`imported_legacy` 及一切 migration provenance 分支 |
| SQLite schema version ledger / fresh baseline DDL 演进 | v1→v2 业务数据 migration、legacy direct-child reconciliation |

### 4. legacy direct-child provenance / reconciliation 全部取消

- 不再为历史 `parent_task_id!=null` 记录编写可读/标 provenance/迁移/补齐流程。
- `TaskCreationProvenance` 生产闭集只保留两支：`root_command_v2`、`child_action_v2`。
- 不得为“让旧库看起来合法”而伪造 Action、Approval、PermissionDecision、Verification 或认证证据；旧库直接拒绝启动。

### 5. Publisher 与 event.poll v2 不在本里程碑

- durable Outbox **写入** 与异步 **Publisher 发布** 是两个责任；本里程碑只要求有业务 owner 的 active Event producer 把事实写入 Outbox。
- retained `EventPollResponse` v1 **不得** 返回 EventEnvelope v2。
- Publisher loop、versioned KCP poll response/binding/handler 另立后续里程碑，不得并入 `V2InitialBuildActive` 完成定义。

### 6. `V2InitialBuildActive` 完成定义

全部为真时，本里程碑完成：

1. **root `task.create` v2** 是唯一 active 写路径：repository + handler + method-aware preflight + production bindings；
2. **child** 唯一经 `task.child.create` Action materializer；
3. **Action / PermissionDecision / Approval v2** 真实持久化闭环（含 current-head CAS、必要 identity/challenge 消费边界按既有合同）；
4. 有业务 owner 的 active Event producer 接入：root/child 的 `task.created`、`action.state_changed`、`approval.state_changed`；
5. 不得伪造无 owner 的 producer；不得启用 Publisher/poll v2 冒充本里程碑完成；
6. v1 runtime 写路径已删除；旧开发库启动拒绝且返回稳定 `reinitialize-required` 诊断；
7. production bindings 口径：`task.create` `active=[2]` 且 v1 仅 `unsupported_schema_version`；其余七方法 `active=[1]`。

## Supersede 范围

本 ADR **部分 supersede** 下列条款的 **生产效力**。被 supersede 的历史正文保留不改写，仅失去“必须实现的生产要求”地位：

| 来源 | 被 supersede 的生产要求 |
|---|---|
| ADR-0006 §7 与 migration/provenance 要求 | legacy direct-child 可读/标 `legacy_direct_create_v1`/`imported_legacy`；migration 不伪造历史但仍须迁移的义务 |
| ADR-0006 验收第 6 条 | legacy direct-child 可读且标记 provenance |
| ADR-0006 KCP 段中 `V2ProductionWriteCutover` 启用叙述 | 以 cutover 作为 production bindings 唯一启用门 |
| ADR-0007 中 v1 migration 生产要求 | production write 前必须完成 v1→v2 Approval/PD migration；repository `legacy v1 read projection` 作为迁移配套生产义务 |
| ADR-0007 实现影响中的 migration 要求 | 同上 |
| ADR-0008 §7 | migration 0003 mixed transform 作为“兼容既有 v1 Outbox 生产库并继续服务”的生产目的；旧库自动升级为 mixed 后继续运行 |
| ADR-0008 §8 | `PendingLegacyEventV1` / `append_legacy_event_v1` / `StoredEventEnvelope::LegacyV1` 作为 production store 长期 API；cutover write gate 叙事 |
| IC 旧 §13.7 `V2ProductionWriteCutover` | 整节谓词与 cutover 前后行为 |
| IC 旧 §6.10.6 migration lifecycle、§6.16.1 legacy provenance 分支 | 迁移生命周期与 legacy provenance kinds |
| CORE §9.2 legacy direct-child 规则、CONFORMANCE 对应测试要求 | 同上 |

**不被 supersede 的部分**（示例）：

- ADR-0006 的 root-only v2、`task.child.create` 唯一 child 入口、atomic materialization、CausationRef v2；
- ADR-0007 的 Approval v2 不可变联合、current-head CAS、fingerprint 分离、身份证据；
- ADR-0008 的 Event v2 八 Schema、Catalog 权威、active Outbox 写入语义、poll v1 不得返回 v2。

## 旧库拒绝策略

| 检测 | 行为 |
|---|---|
| 数据库 schema/ledger/shape 不属于 v2 fresh baseline 可识别集合 | 拒绝 open/start；稳定错误，语义为 reinitialize-required |
| 存在仅能由已删除 v1 write 路径产生、且 v2 baseline 不承认的业务表/行约定 | 同上 |
| 操作员希望继续 | 只能外部备份后 **删除/重建** 数据库文件；Kernel 不提供自动清库或 in-place 数据迁移 |

错误必须可诊断（稳定 code + 不泄漏 SQL/payload 的说明），不得 silent fallback 到只读兼容模式。

实现顺序见 [`docs/PROGRESS.md`](../docs/PROGRESS.md) 中 `V2InitialBuildActive` 七切片；本 ADR 只锁定目标架构，不在本文维护字段级完成清单。

## 验收

1. 规范与 ADR 索引中，`V2ProductionWriteCutover` 不再作为有效生产里程碑；仅允许出现在 ADR-0009 的作废说明或历史 superseded 引用；
2. `TaskCreationProvenance` 生产闭集只有 `root_command_v2` / `child_action_v2`；
3. 无 legacy direct-child migration/reconciliation 生产要求；
4. production bindings / v2 dispatcher / v2 repository 被定义为初始交付，而非 cutover 结果；
5. 旧开发库拒绝启动的 reinitialize-required 策略写死；
6. Publisher 与 event.poll v2 明确不在本里程碑；
7. MATRIX + PROGRESS 唯一进度源与本 ADR 一致。
