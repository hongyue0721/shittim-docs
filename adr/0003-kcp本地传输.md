# ADR-0003：KCP 本地传输

- 状态：accepted
- 日期：2026-07-16

## 背景

Kernel Control Protocol 需要服务 desktop-client、agent-runtime 与本地内部客户端。第一版不建立 Owner 认证，也不应在没有需求时暴露网络端口。领域 KCP 已在规范中定义，但需要选择本地 transport 和 framing。

## 决策

1. 第一版仅提供本地 IPC：
   - Unix 平台使用 Unix Domain Socket；
   - Windows 使用 Named Pipe。
2. 每个连接传输 UTF-8 JSON 文档，使用 **4-byte unsigned big-endian length prefix** framing；长度是随后 JSON bytes 数，不包括 prefix。最大 frame 大小必须由配置和 Schema 校验共同限制，超限返回/记录 `invalid_request` 并关闭该异常连接。
3. 一个连接允许多个按 `request_id` 关联的请求/响应；是否并发处理由实现决定，但同一 Command 的幂等和 revision 语义不因 transport 改变。
4. Event 首批使用 `event.subscribe` + `event.poll` 长轮询，不额外发明 server-push frame 类型。长轮询受 Envelope deadline 约束。
5. Socket/Named Pipe 的文件系统 ACL、owner/peer credential 只作为真实 transport evidence 记录进连接上下文；第一版 Envelope `auth` 仍必须为 null，这些证据不构成唯一 Owner 或默认授权。
6. KCP 消息是项目自有版本化 Envelope，**不采用 JSON-RPC 作为协议事实**。不得要求 `jsonrpc`、`method`、`params` 等 JSON-RPC 字段，也不得把 Extension RPC 的“JSON-RPC 风格”复制到 KCP。
7. TCP、HTTP、WebSocket 与远程 KCP 不在第一版范围。未来引入需要独立 ADR、认证/TLS/速率限制和安全测试。

## 备选方案

- JSON Lines：拒绝，payload 内换行和大消息边界处理更脆弱。
- JSON-RPC 2.0：暂不采用；KCP 已有 actor、`entry_point`、deadline、idempotency、revision 和类型化 payload 语义，强行映射会形成双重 Envelope。
- 本地 HTTP：拒绝，增加端口、HTTP 栈与误暴露面，没有首批收益。
- server-push Event frame：暂缓，首批 poll 足以验证 Outbox cursor 契约。

## 影响

- Unix/Windows 必须共享同一 KCP Schema 和 conformance fixtures。
- transport 实现尚未开始；路径、pipe 名、ACL 默认值和最大 frame 数值由实现配置与测试确定，不能在代码前伪造。
